# 如何让 AI 100% 生成可靠的生产级代码

---

## 一、人是主厨，AI 是副厨

### AI 能写代码，不能为代码负责

在2026 年，任意一个 AI 编码工具都能在 10 秒内生成一个 CRUD 接口。但"能生成"和"能上线"之间，隔着一条鸿沟：**责任归属**。

AI 不会在凌晨三点被报警电话叫醒，不会在代码审查时解释"这个地方没想到"，更不会因为一个内存泄漏导致用户流失而承担责任。**对代码质量负最终责任的，只能是写 Prompt 的那个人。**

这里隐藏着一个核心隐喻：**人是主厨，AI 是副厨。** 副厨可以切菜、备料、甚至炒几道菜，但端上桌之前，主厨必须尝过每一口。不是不信任副厨——是因为主厨的名字写在菜单上。

### 每一行都要审查，没有例外

> "AI 写的代码看起来很对"是最危险的幻觉。

在 Luma 项目中，我给团队定了一条铁律：**AI 生成的每一行代码，必须经过人工审查才能合入主分支。** 这不是效率的倒退——正好相反。AI 也会偷懒——该处理边界条件时它写了个 `// TODO`，该拆分函数时它堆了 200 行，该加错误处理时它假装异常不会发生。这和人类开发者的坏习惯一模一样。区别在于，同事偷懒你可以对他发火，AI 偷懒你只能打回去重写。

审查的标准只有一个：**这是我要上线的项目，每一行烂代码最终都是我来扛。** 风格问题让 Linter 拦，逻辑问题让测试拦，但架构上的"看起来能跑但三个月后会炸"——只有人能在审查时嗅出来。

实操规则：
- AI 提交的 PR 必须附带接收到的 **原始 Prompt**，写在 PR description 里
- 审查时对照 Prompt 和实现，只判断"AI 是否理解错了需求"
- 代码风格、格式化等低级问题由 Lint/Format 工具自动拦下，不消耗人的注意力

### 全栈开发，减少沟通损耗

在团队开发中，一个容易被忽视的事实：**当编码不再是瓶颈时，沟通成为新的瓶颈。**

传统团队里，前端等后端接口、后端等产品确认、产品等 QA 出测试报告——每个环节都在发生信息损耗。AI 让单兵作战能力指数级增长，但前提是**这个人能串起全链路**。

全栈的意义不是贴标签，而是让全栈工程师对上下游有足够的判断力。当一个人既能写前端组件，又知道后端 API 怎么设计最合理时，Prompt 天然少了歧义——不需要向前端同事解释需求，因为自己就是那个同事。

### 判断力是 AI 时代最稀缺的资源

AI 可以替代经验中的"记忆"部分——不再需要记住 API 参数顺序、不再需要记住框架语法。但它替代不了**判断力**：

- 这个架构决策会影响三年后的扩展性吗？
- 这个第三方依赖明天还会维护吗？
- 这段代码的性能瓶颈在哪里——AI 不会主动指出

我的体会是：**每到一个关键节点，人的判断力必须介入。** 选型阶段，AI 可以列出所有方案的优缺点，但最终选 Go 还是 Rust、选单体还是微服务，这个决定只能人来做。AI 是工具，不是替代品。主厨的舌头，永远比副厨的刀更重要。
